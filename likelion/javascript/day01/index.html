자바스크립트의 환경

  1. 브라우저 환경
    • 브라우저에서 살고 있는 자스
  2. 노드 환경
    • 컴퓨터 환경
    • 자스로 서버를 돌릴 수 없을까? 브라우저에만 의존하지 않고? 해서 나온 node
  3. 환경에 따라 사용 방법이 달라지니 구분 필요
    • 브라우저 js와 노드 js의 문법이 다름
    • vite와 같은 도구의 환경설정을 다루려면 node가 꼭 필요함
    • 번들러 : 압축해주는 도구 (환경 세팅에 내용 있음) - 내가 직접 css, img 하나씩 압축하기 넘 귀찮음
    • 번들러 종류 : 웹팩, vite 등
    • 도구를 많이 사용해야 하는 시대에 살고있삼
    • 컴포넌트, 모듈 별로 분류함 (캡쳐참고)
    • 따라서 파일이 많아지니 압축이 꼭 필요함
  4. 환경 세팅
    • 패키지 설치 : npm init -y
    • 패키지 제이슨 : 중괄호 놔두고 다 지우기
    • 라이브 서버 패키지 설치 : npm install live-server
    • 라이브 서버 패키지 제거 : npm remove live-server
    • 옵션 : i (install 약자)
    • 옵션 : -D (devDependencies 약자)
    • dev디펜던시 : 클라이언트(사용자)에게 필요 없는 도구이기 때문에, dev 옵션을 걸어
    • 그냥 디펜던시 예시 : gsap, react, jQuery
    • html,css,js -- build, bundle (압축) --> 서버 업로드 (사용자 접근 가능)
    • 이때 개발종속성(div디펜던시)라면 빌드에 포함이 되지 않아서 용량 크게 압축 가능
    • dev랑 그냥 디펜던시 차이 : 빌드시 용량이 높아짐 > 트래픽 사용량 증가
    • 설치해도 디펜던시에 추가 안된다면 수정중이라 그런것 ! 파일끄거나 저장된 상태에서 재설치 고
    • vs 라이브서버 안쓰고 왜 npm으로 라이브서버를 또 깔아?
    • vs 확장자 라이브서버를 설치하지 않은 사람은 그 화면을 볼 수 없음
    • 내가 사용하고 있는 환경을 설명서(장부)에 기록해두어 다른 사람이 참고할 수 있도록
    • 따라서, 남의 파일 깃에서 클론한다면, npm i 먼저 해야함 !!! (패키지 제이슨 모든 내용 한번에 일괄 설치됨)
    • 내가 설치한 패키지는 모두 디펜던시로 기록이 남음
    • 남의 파일을 가져올 때도 npm i 로 일괄 설치해서 사용해야 함
    • npx live-server client(폴더명) : 패키지를 설치해서 라이브 서버 열었음
    • ^C : control + C 서버 끄기
    • vs liveserver : localhost 뭐시기 ~ 이건 vs liveserver 옵션값
    • 옵션 설정 : npx live-server client --host=localhost --port=5500 --no-browser
    • no-browser 옵션 떄문에 아무것도 안떠서 터미널에 뜬 주소 컨트롤 누르고 클릭해야 함
    • 왜 브라우저 안띄우냐? 서버 여러개 작업하면 개많이 뜰 수 있음
    • 옵션은 npm에서 확인 가능
  5. npx
    • 노드 패키지 실행 (익스큐션)
    • npm 설치로 책을 빌렸어도 펼치려면 npx가 필요함
    • 타인을 위해 이 방법이 더 권장됨
    • npx를 사용하지 않아도 실행시킬 수 있는 방법
    • 도서관에서 빌리지 않고 아예 사버림
    • npm i -g live-server : -g - global
    • -g : global - 빌리는게 아니라 내 컴퓨터에 강제로 설치
    • 하지만 좋은 방법은 아님 - 디펀던시에 표시되지 않기 때문에 다른 사람 컴퓨터에 설치가 안된 경우 알 수가 없음
    • 진짜 누구나 다 ~~ 설치한다면, 전역으로 설치
  6. json 세팅하기
    • json은 쌍따옴표만 사용 가능함 !
    • npx 명령어 제외한 라이브서버 실행 명령어를 script - dev에 작성
    • npm run dev : dev에 입력한거 알아서 자동 실행됨
    • dev 자리에 start, stop, test 등이 올 수 있는데 이 3개는 run 생략 가능
    • run : npx 역할을 함
    • 하나의 스크립트에는 하나의 값(라이브서버)만 지정 가능하고 여러개 작성하려면 쉼표
    • 하지만 쉼표로 끝나면 안됨 아래 바로 다른 스크립트를 작성해야함
    • npm-run-all : 패키지들의 일괄 실행을 위한 패키지
    • 수업땐 무조건 npm run dev !
  7. node로 실행하기
    • server/index.js
    • node server/index.js
    • 근데 왜 이걸 노드로 열어야 하는디?
    • vite : 서버 환경 구성 할 때 내가 작성해야함 - 사실 구문이 거의 비슷해 ~
    • 나중에 보다 깊은 설정은 npm보단 노드로 실행하는 것이 더 편할 수 있음
  8. package-lock.json
    • 패키지 백업용 히스토리
    • 계속 자동으로 백업됨
  9. 큰그림 그리기
    • node : 내 컴퓨터에 설치함. 설치하면 노드 세상의 모든 도구(다양한 개발자들이 만든 패키지)를 사용할 수 있게 됨
    • node의 모든 도구(패키지) : npm을 통해 아무나 패키지를 올리고 사용할 수 있음
    • 패키지 도서관에서 내 컴퓨터로 빌려오기 : 장부에 적어서 빌려줌 - 이때 장부 : 디펜던시스 = 기록용

ESlint와 prettier

  1. ESlint
    • lint : 옷에 붙은 보풀을 제거 - linting 보풀 제거
    • ESlint : ES를 깔끔하게 처리해주는 것
    • 나의 잘못, 실수를 vs코드 자체에서 경고로 알려줌 (콘솔까지 가지 않아도)
    • 개발자 경험을 향상시키는데 상당히 중요함
    • live-server와 동일하게 타 사용자를 위해 장부에 ESlint를 꼭 작성해야함
    • 모든 팀원의 원활한 협업을 위해
  2. ESlint 설치 방법
    • vscode eslint 설치 후 터미널 과정 진행
    • npm init @eslint/config
    • 1. 문법 검사만 (2. 문법&문제점) 3. 문법이랑 문제점 다 찾아서 코드 스타일 강제 변경
    • 3번은 프리티어와 충돌이 발생함
    • 자스 일반 모듈 선택
    • 브라우저, 노드 스페이스바로 2개 선택
    • config파일 형식은? .js (config 등 환경설정 파일은 보통 .으로 파일명이 시작함 = 숨김파일)
    • 설치 지금 하고, 뭘로 설치해? npm
    • 설치 완료하면, .eslintrc.js 파일 생성됨
    • 만약, 선택 잘못했다면 파일 삭제하고 다시 만들면 됨 !
    • 이거 외울필요 없고, ESlint 공식문서에 내용이 나와있음
    • 설정하면 해당 프로젝트에만 적용되고, 프젝 할때마다 번거롭더라도 깔아야함
    • error lens 확장자 함께 사용하면 에러 문구까지 뜸
    • "extends": "eslint:recommended",
    • // eslint가 추천한 항목을 자동 설치
    • 설치할 수 있는 항목이 정말 많다보니, 대화형식으로 다운로드, 그리고 추천항목 자동설치함
  3. ESlint 실행
    • npx eslint ./client/main.js
    • 화면에 나오는데 뭐하러 eslint를 실행해?
    • 실행을 하지 않아도 화면에 나오는 건 vs의 확장자 eslint 덕분
    • 그래서 vs eslint도 같이 깔고 써 ~
  4. 특정 오류 수정
    • /* eslint no-undef:off */ === 0
    • 주석 (//은 안됨) 안에 오류 메시지를 끌 애 작성
    • /* eslint no-undef:warn */ === 1
    • 주의로 변경
    • off = 0, warn = 1, 경고 = 2
    • /* eslint-disable */
    • 해당 파일의 모든 경고 사용안함
    • 변수 사용 안했어 오류는 끄는 것이 편함
    • 모든 파일에 대한 오류 설정은 eslintrc.js에서 일괄 수정
    • "rules": { "no-unused-vars": "off" }
  5. Prettier
    • eslint : 오류 찾기
    • prettier : 코드 스타일 강제
    • 보통 세트로 다님
  6. Prettier 설치 및 환경구성 파일 작성
    • npm i -D prettier
    • .prettierrc.js에 노션 내용 참고하여 작성
  7. Prettier 실행
    • ESlint와 동일하게 매번 명령어를 작성해야함
    • 그래서 vs 확장자 prettier를 함께 설치하면 저장할때마다 실행됨

  2. 모던 자바스크립트
    • 구버전 / 신버전(모던)
    • 구버전 : 3버전 이하
    • 모던 : 5버전 이후
  3. 동일 출처 정책
